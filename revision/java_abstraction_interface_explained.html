Abstraction = hiding HOW, showing WHAT.

Think about driving a car. You press the accelerator and the car moves. You don't know — and don't need to know — how the fuel injectors fire, how the pistons move, how the gearbox shifts. All of that complexity is hidden. You only see the simple interface: accelerator, brake, steering wheel.

That's abstraction. In Java, it means: define what an object should do — but hide the implementation details of how it does it.
In Java, abstraction is achieved in two ways:
1. Abstract classes — a class that can't be instantiated, forces subclasses to fill in details.
2. Interfaces — a pure contract. Just method names, no code. Every implementer must provide the code.
1You define a method name and what it should do — but leave the body empty.
2Every subclass or implementer is forced to provide the actual code.
3The person using your class doesn't need to know HOW it works — just THAT it works.
An abstract class is a half-built class. It has some methods already written (concrete methods), and some methods that are intentionally left empty (abstract methods) for subclasses to fill in.

You cannot create an object directly from an abstract class. It exists only to be extended.
Real world analogy: Think of an abstract class like a house blueprint that already has the walls and roof drawn (concrete methods), but has blank boxes labelled "design this room yourself" (abstract methods). Every house built from that blueprint must fill in those blank rooms. But you can't live in the blueprint itself — you need an actual house.
// Abstract class — half built. Can't do: new Shape() abstract class Shape { // Abstract method — no body, no code. Must be overridden. abstract double area(); abstract double perimeter(); // Concrete method — already implemented. Subclasses inherit this free. void describe() { System.out.println("Area: " + area() + ", Perimeter: " + perimeter()); } } // Subclass MUST implement area() and perimeter() — or it won't compile class Circle extends Shape { double radius; Circle(double r) { this.radius = r; } double area() { return 3.14 * radius * radius; } double perimeter() { return 2 * 3.14 * radius; } } class Rectangle extends Shape { double w, h; Rectangle(double w, double h) { this.w = w; this.h = h; } double area() { return w * h; } double perimeter() { return 2 * (w + h); } } // Usage Shape s = new Circle(5); s.describe(); // Area: 78.5, Perimeter: 31.4 — uses Circle's area() // Shape s = new Shape(); ← COMPILE ERROR. Can't instantiate abstract class!
Rules for abstract classes:
1Must use the abstract keyword on the class.
2Can have both abstract methods (no body) AND concrete methods (with body).
3Can have instance variables (fields like String name, int id).
4Can have a constructor — called via super() from subclass.
5Cannot be instantiated — new Shape() is a compile error.
6A class can extend only ONE abstract class (single inheritance).
When to use abstract class: when your classes are closely related and share common code. Shape, Circle, Rectangle are all related geometric shapes. It makes sense for Shape to provide the describe() method free — all shapes use it the same way.
An interface is a pure contract. It says: "if you want to be called X, you MUST be able to do these things." It provides zero implementation — just a list of method signatures. Every class that implements it must write all the code.

The key word is UNRELATED. An interface lets completely unrelated classes share a common behaviour without being in the same family tree.
Real world analogy: Think of a job description. It says: "To be hired as a Chef, you must be able to: cook(), plateFood(), managekitchen()." It doesn't tell you HOW to cook. A person, a robot, or a trained monkey can all be a Chef — as long as they can do those things. The job description is the interface. Anyone who meets the requirements gets the title.
// Define the interface — just method names, no code interface Flyable { void fly(); // no body — just a contract int getAltitude(); // every implementer must write this } // Three UNRELATED classes — all implement Flyable class Bird implements Flyable { public void fly() { System.out.println("Flapping wings!"); } public int getAltitude() { return 50; } } class Airplane implements Flyable { public void fly() { System.out.println("Engines at full throttle!"); } public int getAltitude() { return 35000; } } // One class can implement MULTIPLE interfaces — this is the superpower interface Swimmable { void swim(); } class Duck implements Flyable, Swimmable { public void fly() { System.out.println("Duck flying!"); } public int getAltitude() { return 10; } public void swim() { System.out.println("Duck swimming!"); } } // The magic: one method works for ALL Flyable objects void launchIntoAir(Flyable f) { f.fly(); System.out.println("At altitude: " + f.getAltitude() + "ft"); } launchIntoAir(new Bird()); // Flapping wings! At altitude: 50ft launchIntoAir(new Airplane()); // Engines at full throttle! At altitude: 35000ft
The interface superpower: A class can implement multiple interfaces, but only extend one class. So interfaces are how Java achieves multiple inheritance of behaviour — a Duck can be both Flyable AND Swimmable.
Rules for interfaces:
1Use the interface keyword, and implements in the class.
2Before Java 8: all methods are abstract (no body). Java 8+: can have default and static methods with a body.
3No instance variables — only public static final constants.
4No constructors.
5A class can implement multiple interfaces — this is the big difference from abstract class.
6All methods in an interface are implicitly public — you must use public when implementing them.
Both abstract class and interface achieve abstraction. But they're used for different situations. Here's how to think about which to use — and how to answer this in an interview.
Abstract class
✓Can have concrete methods (with code)
✓Can have instance variables
✓Can have constructors
✓Methods can be any access modifier
✗One per class (no multiple inheritance)
✗Can't be instantiated
Interface
✓Pure contract — only method signatures
✓Multiple interfaces per class
✓Java 8+: default & static methods
✗No instance variables
✗No constructors
✗All methods implicitly public
How to decide which one to use — the decision rule:
Use abstract class when: The classes are in the same family. They share real code. It's an IS-A relationship.
Example: Shape → Circle, Rectangle, Triangle. All shapes. All share the describe() method. Makes sense to be one family.
Use interface when: Unrelated classes need to share a behaviour. It's a CAN-DO relationship.
Example: Bird, Airplane, Superman can all fly — but they're completely unrelated. Use Flyable interface. A Duck can be Flyable AND Swimmable — impossible with abstract class.
// Abstract class — IS-A relationship, shared family abstract class Vehicle { String brand; // shared field — all vehicles have a brand void startEngine() { ... } // shared method — all vehicles start the same way abstract void move(); // different for car vs boat vs plane } // Interface — CAN-DO capability, unrelated classes interface GPS { void getLocation(); // a Car, Phone, or Watch can all have GPS } class Car extends Vehicle implements GPS { void move() { System.out.println("Driving..."); } public void getLocation() { System.out.println("28.6N 77.2E"); } } // Car IS-A Vehicle (abstract class) AND CAN-DO GPS (interface)
Interview answer template: "Abstract class is used for IS-A relationships among closely related classes where shared code exists. Interface is used for CAN-DO capabilities that unrelated classes can share. The key practical difference is that a class can implement multiple interfaces but extend only one abstract class."
