What is Spring? Start here.

Before touching any code, understand WHY Spring exists and what problem it solves.

Imagine you're building a restaurant. You need a chef, a waiter, and a cashier. Each of these people needs tools — the chef needs a knife and stove, the waiter needs an order pad, the cashier needs a register.

Without Spring: you (the programmer) manually create every person, buy them every tool, and wire them all together. If the chef needs a knife, you write: Chef chef = new Chef(new Knife(), new Stove()).

With Spring: Spring is the restaurant manager. You just say "I need a Chef." Spring knows a Chef needs a Knife and Stove, creates all of them, wires them together, and hands you a fully ready Chef. You don't worry about how.
Three things Spring gives you:
1IoC Container (Inversion of Control) — Instead of you creating objects, you hand control over to Spring. Spring creates and manages all your objects (called Beans). This is the core idea.
2Dependency Injection — Spring automatically provides (injects) everything an object needs. A Chef that needs a Knife gets one automatically — you just declare the dependency.
3Modules — Spring has modules for web (Spring MVC), databases (Spring Data), security (Spring Security), etc. You pick what you need.
One sentence answer for the interview: "Spring is a Java framework that manages object creation and wiring through Inversion of Control and Dependency Injection, so developers focus on business logic instead of boilerplate setup code."
What is a Bean?
A Bean is simply any object that Spring creates and manages. When you mark a class with @Component or @Service, you're telling Spring: "please create and manage an object of this class." That managed object is called a Bean. Spring stores all Beans in its container (called the ApplicationContext) and injects them wherever needed.

Dependency Injection (DI)

The most important concept in Spring. Understand this and everything else falls into place.

Dependency Injection means: instead of an object creating what it needs, someone else provides it.

If a Chef needs a Knife, there are two ways:
— Without DI: Chef itself creates a new Knife() inside its own constructor. Tightly coupled. Hard to test.
— With DI: Someone from outside hands the Knife to the Chef. The Chef doesn't care WHO gives it the knife — just that it gets one. Spring is that "someone."
Real world analogy: You go to a hospital. You don't carry your own blood pressure machine, stethoscope, and medicine. The hospital provides (injects) everything the doctor needs. The doctor just does their job. That's DI — the professional gets what they need, provided by the environment, not by themselves.
// WITHOUT Dependency Injection — tightly coupled, hard to test class Chef { private Knife knife; Chef() { this.knife = new Knife(); // Chef is stuck with THIS specific Knife forever } // Can't swap it. Can't test with a fake knife. } // WITH Dependency Injection — loosely coupled, easy to test class Chef { private Knife knife; Chef(Knife knife) { // Knife is GIVEN to Chef from outside this.knife = knife; // Chef doesn't care where it came from } } // Spring does this automatically — you never write "new" for your services
The 3 types of Dependency Injection in Spring
1 Constructor Injection — dependencies passed through the constructor. This is the recommended way. It makes dependencies clear and the object is always fully initialized.
@Autowired Chef(Knife knife) { this.knife = knife; }
2 Setter Injection — Spring calls a setter method to provide the dependency. Used when the dependency is optional.
@Autowired void setKnife(Knife knife) { this.knife = knife; }
3 Field Injection — put @Autowired directly on the field. Simplest to write but least recommended (hard to test).
@Autowired private Knife knife;
// Constructor Injection — RECOMMENDED @Service class ChefService { private final KnifeService knifeService; @Autowired // Spring injects KnifeService here automatically ChefService(KnifeService knifeService) { this.knifeService = knifeService; } } // Field Injection — common in tutorials, but less ideal @Service class ChefService { @Autowired private KnifeService knifeService; // Spring injects this field directly }
Why DI matters for interviews: It makes code loosely coupled (easy to change one part without breaking others), testable (you can inject a fake/mock object for testing), and maintainable. These are the exact words interviewers want to hear.

Spring annotations — the most asked topic

Annotations are instructions you give Spring by putting @Something above your class or method. Know these 10 cold.

An annotation is just a label. When you write @Service above a class, you're telling Spring: "this class is a service — please manage it as a Bean." Spring reads these annotations when the app starts and sets everything up automatically.
See them all working together:
// A complete mini Spring application — 3 layers // 1. Repository layer — talks to database @Repository public class UserRepository { public User findById(int id) { /* DB query */ return user; } } // 2. Service layer — business logic @Service public class UserService { @Autowired private UserRepository userRepo; // Spring injects this public User getUser(int id) { return userRepo.findById(id); // business logic here } } // 3. Controller layer — handles HTTP requests @RestController @RequestMapping("/users") public class UserController { @Autowired private UserService userService; // Spring injects this @GetMapping("/{id}") public User getUser(@PathVariable int id) { return userService.getUser(id); // delegates to service } }

Spring Boot — Spring but easier

Spring Boot is not a replacement for Spring — it's Spring with the setup already done for you.

Setting up a Spring application from scratch used to require writing hundreds of lines of configuration — XML files, setting up a web server, configuring databases, connecting everything. It took hours just to say "hello world."

Spring Boot removes ALL of that setup. It provides smart defaults, auto-configuration, and an embedded web server. You go from zero to a running web app in minutes.
Analogy: Spring is like buying land and building a house from scratch — choosing bricks, laying foundation, wiring electricity. Spring Boot is like moving into a furnished apartment — everything is already set up, you just bring your stuff (your business logic) and start living.
1Auto-configuration — Spring Boot detects what libraries you have and configures them automatically. Add a database driver? Spring Boot automatically sets up the connection pool. Add a web library? It starts a web server. You don't write any config.
2Embedded server — Spring Boot includes a Tomcat web server inside your app. You just run your Java program like any normal main() method — no separate server installation needed.
3Starter dependencies — Instead of hunting for 10 individual libraries, you add one "starter" and Spring Boot pulls in everything you need. spring-boot-starter-web gives you web + JSON + Tomcat in one line.
4Production-ready — Spring Boot Actuator gives you health checks, metrics, and monitoring out of the box at /actuator/health.
// A complete Spring Boot REST app in ~15 lines @SpringBootApplication // enables everything — component scan + auto-config public class MyApp { public static void main(String[] args) { SpringApplication.run(MyApp.class, args); // starts the embedded server } } @RestController public class HelloController { @GetMapping("/hello") public String hello() { return "Hello World!"; // visit localhost:8080/hello and see this } } // That's it. A fully running web server serving HTTP requests. // No XML. No server setup. No configuration files.
Interview one-liner: "Spring Boot is an opinionated extension of Spring that provides auto-configuration, embedded servers, and starter dependencies to eliminate boilerplate setup. It lets you build production-ready Spring applications with minimal configuration."
What @SpringBootApplication actually does:
1@SpringBootConfiguration — marks this as a Spring configuration class (like @Configuration).
2@EnableAutoConfiguration — tells Spring Boot to automatically configure everything based on what's in your classpath.
3@ComponentScan — scans the current package and all sub-packages for @Component, @Service, @Controller, @Repository classes and registers them as Beans.
@SpringBootApplication = @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan. That's the most common "trick" interview question about this annotation.

Building REST APIs with Spring Boot

This is what you'll most likely do day-to-day. Understand every line of this.

A REST API is a way for applications to communicate over HTTP. Your Spring Boot app exposes URLs (endpoints) that other apps or frontends can call. Each URL does something: return data, create something, update something, delete something.

HTTP methods map to actions: GET = read, POST = create, PUT = update, DELETE = delete.
// Full CRUD REST API for a User resource @RestController @RequestMapping("/api/users") public class UserController { @Autowired private UserService userService; // GET /api/users — return all users @GetMapping public List<User> getAllUsers() { return userService.getAllUsers(); } // GET /api/users/5 — return user with id 5 @GetMapping("/{id}") public User getUserById(@PathVariable int id) { return userService.getUser(id); } // POST /api/users — create a new user @PostMapping public User createUser(@RequestBody User user) { return userService.createUser(user); // user data comes in as JSON } // PUT /api/users/5 — update user with id 5 @PutMapping("/{id}") public User updateUser(@PathVariable int id, @RequestBody User user) { return userService.updateUser(id, user); } // DELETE /api/users/5 — delete user with id 5 @DeleteMapping("/{id}") public void deleteUser(@PathVariable int id) { userService.deleteUser(id); } }
@PathVariable vs @RequestParam: @PathVariable pulls from the URL path: /users/5 → id=5. @RequestParam pulls from query string: /users?name=Aneesh → name="Aneesh".
@RequestBody: When someone sends JSON data in a POST/PUT request, @RequestBody tells Spring to automatically convert that JSON into a Java object. Spring uses Jackson (a JSON library) to do this automatically.
