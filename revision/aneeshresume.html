FedEx internship — complete walkthrough

Learn this step by step. You should be able to explain it like you're walking someone through your own kitchen.

The story starts with a problem, not with the technology. Always open with: "The team had a problem — they were spending 8 hours every week doing something manually."
1What was the situation / problem?
The FedEx operations team at the Hyderabad distribution hub was managing logistics data across multiple distribution hubs — tracking worker productivity, how efficiently they were using their time, how many packages were being processed, etc.

The problem: All of this data was sitting in raw files (spreadsheets, system exports). Every week, someone had to manually pull that data, clean it, combine it, and format it into a report. That was 8 hours of work every single week — wasted on a repetitive task that a machine should be doing.

Why it mattered: By the time the manual report was done, the data was already old. Senior stakeholders were making decisions based on last week's numbers, not today's reality.
2What was your task?
Your job was to eliminate that manual work completely — build a system where the data flows in, gets cleaned, and turns into a live dashboard automatically, every single day. No human involvement after setup.

You also had to make sure senior stakeholders could read and trust it — so accuracy and clarity were non-negotiable.
3What did you actually build? (Part 1 — the ETL pipeline)
ETL = Extract, Transform, Load. This is the backbone — getting raw data into a clean, usable form.

You used Power Query M (the programming language built into Power BI's data editor) to:

— Extract: Connect to the raw data sources from multiple distribution hubs
— Transform: Clean inconsistencies (different column names across hubs, missing values, date format mismatches, duplicate records)
— Load: Structure it into a clean, unified table that Power BI could work with

You then scheduled this pipeline to run automatically every day, refreshing 50,000+ records overnight — so when anyone opened the dashboard in the morning, they were seeing today's data, not last week's.
4What did you actually build? (Part 2 — the dashboard)
On top of the clean data, you built the Power BI dashboard using DAX (Data Analysis Expressions) — Power BI's formula language — to calculate the KPIs:

— Productivity Rate: Packages processed per worker per hour
— Utilization %: What percentage of scheduled work hours were actually spent on productive tasks
— Labour Efficiency: Actual output vs expected output

The dashboard showed these metrics in real-time, broken down by hub, by time period, by shift. Stakeholders could filter and drill down without needing any technical knowledge.

Why DAX? Because these KPIs aren't just simple sums — Utilization %, for example, requires dividing productive hours by total hours while handling edge cases like zero values. DAX lets you write that logic once as a "measure" and it recalculates correctly no matter what filter the user applies.
5What were the results?
— Reporting time reduced by 40%
— 8 hours/week saved for the operations team
— 30% reduction in data preparation time
— 50,000+ records refreshed automatically every day
— Presented directly to senior stakeholders who adopted it for operational decision-making
If they ask "what was the hardest part?" — Say: "The biggest challenge was that the raw data wasn't standardized across hubs. Different hubs used different column names and some had missing values. I had to write custom Power Query M transformations that were robust to these variations, and then validate the output by spot-checking the dashboard numbers against manual calculations before presenting to stakeholders."

E-Commerce segmentation — complete walkthrough

This is your most technically impressive project. Focus on the BigQuery pipeline and the RFM insight.

Open with: "This project was about answering a business question — which customers should a company spend its marketing budget on?"
1What was the problem / goal?
E-commerce companies collect massive amounts of web traffic data through Google Analytics 4 (GA4) — every click, every page view, every purchase, every user action is logged as an "event". But this data is extremely messy. It's stored in Google BigQuery in a nested format — imagine a spreadsheet where one cell contains an entire mini-table inside it (that's what nested JSON arrays look like).

The goal: Extract meaningful business metrics from this messy raw data, then use those metrics to segment customers so the marketing team knows exactly who to target.
2The data — what were you working with?
1 million+ raw GA4 event logs stored in Google BigQuery.

GA4 stores event parameters as nested arrays — meaning each row in the table has a column called "event_params" which contains a list of key-value pairs. For example: one event row might have event_params containing [{key: "page_title", value: "Checkout"}, {key: "item_price", value: 1499}, {key: "currency", value: "USD"}] — all packed inside a single cell.

Why is this hard? Normal SQL can't just say SELECT item_price FROM events — because item_price is buried inside that nested structure. You need to "unnest" or flatten it first.
3What did you build? (The BigQuery SQL pipeline)
You used BigQuery SQL with CTEs and UNNEST() to solve this.

UNNEST() is a BigQuery function that takes a nested array and expands it into individual rows — flattening the structure so you can query it normally.

CTEs (WITH clauses) let you break your query into readable, named steps — like Step 1: flatten the data, Step 2: extract the metrics, Step 3: aggregate by user.

Using this approach, you extracted 15+ hidden revenue metrics that weren't visible in the standard GA4 interface — things like: revenue per session, purchase funnel drop-off rates, product-level performance, user-level spending patterns, and multi-session attribution.

Why BigQuery? GA4's built-in reporting interface only gives you pre-built charts. BigQuery gives you raw access to every single event — so you can answer questions that GA4's UI can't answer.
4What is RFM segmentation and what did you do?
RFM stands for:
— R — Recency: How recently did this customer buy? (A customer who bought yesterday is more valuable than one who bought 6 months ago)
— F — Frequency: How often do they buy? (10 purchases vs 1 purchase)
— M — Monetary: How much do they spend in total?

You scored every customer on each dimension (e.g., 1–5 score for each), then combined the scores to put customers into segments:

— Champions: High R, High F, High M — your best customers, buy often, spend a lot, bought recently
— Loyal Customers: Buy often but maybe not the highest spenders
— At-Risk: Used to be good customers but haven't bought recently
— Lost: Low on all three — not worth spending budget on

You used K-Means clustering (a machine learning algorithm) and Scikit-Learn to automate the segmentation across all users.
5What was the insight and business impact?
The key finding: the top 27% of users (Champions + Loyal) were generating the vast majority of revenue. The bottom segments were consuming marketing budget but contributing very little in return.

Business recommendation: Redirect marketing spend toward retaining and upselling the top 27% instead of broad acquisition campaigns. Estimated impact: 35% improvement in marketing efficiency — meaning the same budget produces 35% better results by targeting the right people.
If they ask "what is K-Means clustering?" — Say: "K-Means is an unsupervised machine learning algorithm that groups data points into K clusters based on similarity. I used it here to automatically group customers by their RFM scores — instead of manually setting thresholds, the algorithm found the natural groupings in the data."

Supply chain delay prediction — complete walkthrough

This is your most memorable project because of the counter-intuitive finding. Lead with the surprise, not the accuracy.

Open with: "I wanted to answer one question — can we predict which shipments will be late before they're late? Because catching delays in advance means you can do something about them."
1What was the problem / goal?
Late deliveries are expensive — for the customer (broken trust), for the company (refunds, complaints), and for operations (reactive firefighting). If you could predict which shipments are at risk of being late before they're late, you could change the routing, prioritize them, or set better expectations.

Goal: Build a machine learning model that takes shipment data as input and predicts: will this delivery be late or on-time?
2The data
180,519 supply chain records. Each record had information like: shipping mode (First Class, Second Class, Standard, Same Day), product category, order quantity, customer location, warehouse location, order date, promised delivery date, actual delivery date, and whether it was late or not.

The target variable (what you're predicting) was a binary label: Late = 1, On-Time = 0.

You did EDA (Exploratory Data Analysis) first using Python, Pandas, Seaborn, and Matplotlib — visualizing distributions, checking for nulls, looking at class balance, and plotting relationships between features.
3Why Random Forest? (This question WILL be asked)
Random Forest is an ensemble method — it builds many decision trees on random subsets of data, then combines their votes. You chose it because:

— It handles mixed data types naturally (your data had both categorical columns like shipping mode and numerical columns like quantity)
— It gives you feature importance scores — you can see which variables the model found most useful for predicting delays. This was critical for generating business insights, not just predictions.
— It's robust to outliers and doesn't need feature scaling (no need to normalize data)
— It reduces overfitting compared to a single decision tree, because many trees averaging out is more stable than one tree memorizing the training data
4What did you build and what were the results?
You split the data into training and test sets, trained a Random Forest Classifier using Scikit-Learn, and achieved 70% accuracy on the test set.

Then you used model.feature_importances_ to rank which variables mattered most for predicting delays. Shipping mode came out as the strongest predictor by far — more than location, product type, or order size.

So you drilled in and looked at delay rates per shipping mode.

That's when you found it.
The counter-intuitive finding: First Class shipping — the most expensive, premium option — had a 95% delay rate. This was the complete opposite of what you'd expect. Customers paying the most for the fastest delivery were the most likely to receive it late. This challenged a fundamental assumption about the business — that premium = reliable. The root cause could be over-promising on SLAs, routing inefficiencies, or capacity constraints during peak periods. This finding alone makes the project actionable and memorable.
5What's the business impact?
If deployed in a live system, the model could flag high-risk shipments at the time of order placement — before anything goes wrong. That allows the operations team to:

— Re-route First Class shipments to more reliable carriers or paths
— Proactively communicate delays to customers before they complain
— Adjust SLA commitments for First Class in peak periods

Estimated impact: potential 20% reduction in late deliveries if deployed in a live logistics pipeline.

Global salary dashboard — complete walkthrough

This one is simpler technically — but the business thinking behind it is strong. Lead with the user experience decision.

Open with: "I chose Excel for this deliberately — not because I couldn't use Python, but because Excel was the right tool for the audience."
1What was the problem / goal?
HR teams and job seekers waste a lot of time researching salary benchmarks — manually searching job boards, cross-referencing countries, filtering by role and experience level. There was no single place to instantly see "what does a Senior Data Analyst earn in Germany vs India vs USA?"

Goal: Build a dashboard that answers salary benchmarking questions in one click, with no technical expertise required.
2The data
32,000+ job records across 50+ countries. Each record had: job title, company size, experience level (entry/mid/senior/executive), employment type, employee residence, company location, remote ratio, and salary in USD.

The challenge was that 32,000 rows across 50 countries means the data is inherently multi-dimensional — you need to filter multiple variables simultaneously to get a meaningful answer.
3Why Excel — not Python or Power BI?
This is the most important question for this project. Your answer shows maturity.

The audience was HR teams and hiring managers — people who live in Excel. A Python script requires someone to re-run it. A Power BI dashboard requires a license and setup. But an Excel file? Anyone can open it, filter it, and share it — zero friction.

The goal was adoption — getting people to actually use the tool. Excel was the best tool for that goal.
4What did you build technically?
— Array formulas: Excel formulas that calculate across entire ranges simultaneously — used to compute average salaries for filtered subsets without pivot tables
— Slicers: Visual filter buttons connected to the data — click "Senior" and "Germany" and everything updates instantly
— Data Validation: Dropdown menus for Role, Country, and Experience Level so users can't enter invalid filters
— Boolean logic: Used inside array formulas to apply multiple conditions at once (e.g., where role = "Data Analyst" AND country = "India" AND level = "Senior")
— Statistical analysis: Min, max, median, percentiles per segment — so users could see not just the average salary but the full range
5What was the impact?
The dashboard reduced salary benchmarking research time by 70% — replacing manual lookups that took 30+ minutes with an instant filtered view. Stakeholders could make compensation decisions in minutes instead of hours, with data-backed numbers instead of guesswork.
If they say "this seems simple compared to your other projects" — Say: "That's exactly the point. The FedEx dashboard and BigQuery pipeline were complex because the problems required complexity. This project required simplicity — the problem was that HR teams needed something they'd actually open and use every day, not something technically impressive that sits unused. Knowing when to use Excel vs Python vs Power BI is itself a skill."
